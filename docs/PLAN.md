## 1. 비전

이직 준비 중인 개발자가 매일 채용 공고를 수동 확인하고 공고마다 지원 자료를 처음부터 쓰는 비용을 없앤다. 완성형 그림: 공고가 자동으로 모이고 → 적합도순으로 정렬되고 → 내 경험 기반 초안이 미리 써져 있고 → 알림으로 오고 → 지원 현황까지 관리되는 개인용 지원 전략 OS. 로컬 웹앱으로 시작하고, 잘 되면 서비스화.

## 2. 절대 원칙 — AI 저작권 경계

> "AI가 사람의 인생에 대한 생각과 경험을 창조할 수는 없으니까." (하혜민, 2026-07-07)

AI는 **작가가 아니라 편집자 + 인터뷰어**다. 이 원칙은 3중으로 강제된다:

| 층 | 메커니즘 |
|---|---|
| 스키마 | AI 생성 문장은 `primary_source_card_id` NOT NULL + CHECK 제약 — 출처 없이는 저장 불가 |
| 파이프라인 | 근거 없는 요구사항 → 문장 대신 되묻기(askback) 생성. LLM이 지어낸 카드 ID는 코드에서 차단 |
| 2차 검증 | 생성 문장이 카드의 "주장해도 되는 것" 범위를 넘으면 over_claim 경고 |

최종 게이트는 사용자 검토. 100% 기계적 보장은 불가능함을 인정하고 UI에 경고로 노출한다.

## 3. 왜 이 구조인가 (시장 근거)

- 기존 AI 지원 도구(Simplify 등)의 공통 평가: "이력서 한 장에서 생성 → 너무 제네릭해서 다 고쳐 써야 함"
- 국내 취준 통설: AI가 통째로 쓴 자소서는 감점, AI를 소재·구조 정리 도구로 쓰면 합격률 상승
- 오픈소스 수집기(JobSpy 등)는 전부 해외 잡보드 대상 — 원티드/잡코리아를 다루는 개인 도구는 공백
- **차별점: 경험 뱅크 + 출처 강제 + 되묻기 루프.** 저작권 모델 자체가 다르다

## 4. 데이터 모델 요약

`src/db/schema.ts`가 구현 정본. 핵심 관계:

```
experience_cards (경험 뱅크 — 원본. soft delete만 허용)
  ↑ 출처 참조 (AI 문장은 필수)
job_postings → requirements → matches(근거 강도 강/중/약) → experience_cards
            → drafts → draft_sentences (ai|user|placeholder, 경고, 출처 다대다)
            → askbacks (되묻기 → 답변이 새 카드/카드 보강으로)
            → interview_questions → answer_points (동일한 출처 규칙)
            → applications (지원 이후 결과의 단일 source of truth)
```

- 근거 강도 규칙: 강/중 → 문장 생성, 약 → 생성+경고+되묻기, 없음 → 되묻기만
- 카드 수정 시 해당 카드를 출처로 가진 문장에 "출처 변경됨" 표시

## 5. LLM 전략

- `src/lib/llm.ts` 추상화를 통해서만 호출 (직접 SDK/fetch 금지)
- 기본: **Ollama 로컬 모델** (qwen3:8b) — 무료, 데이터 로컬 유지. `ANTHROPIC_API_KEY` 설정 시 Claude 자동 전환
- 두 경로 모두 zod 스키마 기반 구조화 출력
- 로컬 8B 품질 한계 대응: 프롬프트 단순화 → 호출 분할 → 모델 교체(exaone3.5/qwen3:14b) 순으로 시도

## 6. 단계별 로드맵

| 단계 | 내용 | 상태 |
|---|---|---|
| **1단계: 붙여넣기 코어** | 카드 CRUD, 공고 붙여넣기→분해→매칭→적합도→출처 초안→되묻기→면접 질문 | ✅ 완료 (2026-07-07) |
| **1.5단계: 품질 안정화** | 실사용 피드백 반영, 적합도 인플레이션/면접 질문 수 등 로컬 모델 품질 이슈 | 🔄 진행 중 |
| **2단계: 원티드 수집** | 검색 조건 저장 → 주기 수집 → 같은 DB 유입 + 적합도순 피드 | 대기 (1단계 실사용 검증 후) |
| **3단계: 확장** | 잡코리아/사람인 수집, 링크드인 알림 이메일 파싱(직접 크롤링 금지), 텔레그램 알림, 지원 현황 대시보드+결과 학습 루프, 자소서 문항별 답변 | 대기 |

**순서 규칙: 앞 단계가 실사용에서 검증되기 전에 뒷 단계를 만들지 않는다.**

## 7. 성공 기준

- 1단계: 실제 공고 하나를 붙여넣었을 때, **그날 지원에 바로 쓸 수 있는** 초안+되묻기가 나온다
- 모든 AI 생성 문장에 출처가 달려 있고, 되묻기에 답하면 경험 뱅크가 실제로 자란다
- 2단계: 아침에 열면 밤새 수집된 원티드 공고가 적합도순으로 정렬되어 있다

## 8. 열린 질문

- 적합도 채점 루브릭/가중치 — 실제 공고로 튜닝 (1.5단계)
- 원티드 수집 방법 (비공식 API vs HTML 파싱) — 2단계 시작 시 조사
- 링크드인 이메일 접근 방식 (IMAP/Gmail API/포워딩) — 3단계 전 결정
- 서비스화 시 개인정보 처리 (경험 뱅크는 민감 데이터) — 서비스화 결정 시
